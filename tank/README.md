# Sync Tank Tank Node

Raspberry Pi tank-node software for Sync Tank. The Pi controls local servos, exposes USB camera feeds, receives ESP32-S3 floater images, and forwards selected frames or stream metadata to a separate organizer/main hub computer for VLM/visual processing.

## New Tank / New Raspberry Pi

Recommended fresh Pi setup after cloning this repository:

```bash
cd /home/one/Projects/sync-tank/tank
./scripts/setup-tank.sh
```

The setup wizard asks which Pi/tank this is, which profile it should use, which ESP32 floaters belong to it, how many ReefScopes/Lighthouses/REEFLEX arms/feeders are expected, whether to enable I2C, whether to install required packages, and how it should boot after a power loss.

Recommended profiles are stored in:

```text
config/tank_profiles.yaml
```

Recommended split:

```text
tank1-lighthouse:
  1 feeder, 2 ReefScopes, 2 floaters, 1 Lighthouse, 0 REEFLEX

tank2-reeflex:
  1 feeder, 2 ReefScopes, 2 floaters, 0 Lighthouse, 1 REEFLEX
```

After setup and any required reboot:

```bash
cd /home/one/Projects/sync-tank/tank
./scripts/start-tank.sh
./scripts/preflight-tank.sh
```

If `setup-tank.sh` enabled I2C for the first time, reboot before expecting PCA9685 servo control to work.

Package this tank-node app from a working Pi:

```bash
cd /home/one/Projects/sync-tank/tank
./scripts/package-tank.sh /tmp
```

Copy the generated archive to the new Pi, extract it under `/home/one/Projects/sync-tank/tank`, then run the setup wizard or installer from that directory.

Example for Tank 2 with two ESP32 floaters, two ReefScope cameras, one REEFLEX arm, and one solid feeder:

```bash
cd /home/one/Projects/sync-tank/tank
./scripts/install-tank.sh \
  --node-id tank-pi-002 \
  --label "TANK NODE 2" \
  --profile tank2-reeflex \
  --tank-id tank-2 \
  --tank-label "Tank 2" \
  --expected-nodes tank-cam-003,tank-cam-004 \
  --reefscope-count 2 \
  --lighthouse-count 0 \
  --reeflex-count 1 \
  --solid-feeders 1 \
  --wired-edge-ip TANK_ONE_WIRED_IP \
  --install-boot cron
```

Start the edge node:

```bash
./scripts/start-tank.sh
```

Run the hardware self-test:

```bash
./scripts/self-test-cameras.sh --repair
```

Run the full preflight:

```bash
./scripts/preflight-tank.sh
```

If this Pi is wired directly to a display Pi, configure the static Ethernet link:

```bash
sudo ./scripts/configure_wired_link.sh edge
```

Default wired endpoints for the display Pi:

```text
http://TANK_ONE_WIRED_IP:8080/api/pc-hub/payload
http://TANK_ONE_WIRED_IP:8080/api/hub-payload
http://TANK_ONE_WIRED_IP:5050/api/cameras/usb_0/snapshot
http://TANK_ONE_WIRED_IP:5050/api/cameras/usb_0/stream
```

Servo/control endpoints for the organizer:

```text
GET  http://TANK_ONE_WIRED_IP:5050/api/arm
POST http://TANK_ONE_WIRED_IP:5050/api/servo/channel
POST http://TANK_ONE_WIRED_IP:5050/api/lighthouse/pose
POST http://TANK_ONE_WIRED_IP:5050/api/reeflex/pose
POST http://TANK_ONE_WIRED_IP:5050/api/arm/stop
```

Default PCA9685 channel map:

```text
lighthouse_pan      channel 0
lighthouse_tilt     channel 1
reeflex_base        channel 2
reeflex_shoulder    channel 3
reeflex_elbow       channel 4
```

For real servo movement, enable I2C on the Pi and confirm the configured bus exists:

```bash
ls -l /dev/i2c*
```

The default PCA9685 config uses address `0x40` on bus `1`. If the board appears on a different bus, update `arm.pca9685.bus` in `config/sync_tank.yaml`.

The installer resets runtime state for the new Pi, writes `config/sync_tank.yaml`, writes `config/node_config.json`, and can install a user crontab boot launcher. Use `--install-boot systemd` only when sudo/systemd setup is available.

The canonical Pi identity file is:

```text
config/tank_identity.yaml
```

Use it to declare which tank/node this Pi serves, the wired URLs it should advertise, assigned ESP32 floater IDs, and expected local inventory. `config/sync_tank.yaml` contains service and hardware defaults; `config/node_config.json` is local runtime setup state and is generated by the installer.

Use `config/tank_profiles.yaml` to keep repeated tank setups consistent across Pi nodes.

System packages used on a full Pi install:

```text
python3-pip
python3-venv
ffmpeg
v4l-utils
i2c-tools
python3-smbus
network-manager
curl
```

Python packages are listed in `requirements.txt`.

## ESP32 Camera Upload Receiver

Run the receiver for the 4 sleeping ESP32-S3 camera nodes:

```bash
cd /home/one/Projects/sync-tank/tank
./scripts/run-ingest.sh
```

It listens on `0.0.0.0:8080` and exposes:

```text
POST /api/images/upload
POST /api/node/heartbeat
GET /
GET /uploads/<node_id>/<filename>.jpg
```

In the deployed offline topology, configure each ESP32 Floater to target the private AP address hosted by its owning tank Pi:

```text
Tank One: http://TANK_ONE_AP_IP:8080
Tank Two: http://TANK_TWO_AP_IP:8080

POST /api/images/upload
POST /api/node/heartbeat
GET  /api/node/<camera-id>/command
```

The Floaters do not join the household LAN or contact Sync directly. The tank Pi receives their JPEGs on its `wlan0` AP, then advertises the latest-image URLs to Sync through its Ethernet/PoE connection. See [`../docs/FLOATER_NETWORK.md`](../docs/FLOATER_NETWORK.md) for the complete assignments and addresses.

Images are stored under `test_uploads/<node_id>/`. Receiver state is stored in `config/ingest_state.json`. The dashboard auto-refreshes every 10 seconds at `http://<pi-ip>:8080/`.

The ingest dashboard is intentionally minimal. Use it to confirm this Pi node's local inventory, label cameras, and classify attached sources before forwarding consolidated data to the PC hub.

The Pi node persists local node settings in `config/node_config.json`, including:

- robotic arm count
- endoscope camera count
- floater camera count
- solid/liquid/misc feeder counts
- camera labels and camera types

The PC hub should own final 3D composition and VLM/LLM workflows. The Pi exposes a consolidated payload for the hub at:

```text
GET /api/hub-payload
```

`config/tank_layout.json` is still kept as a basic machine-readable layout asset for hub synchronization, but the 8080 Pi interface should stay focused on tank-node confirmation rather than detailed modeling.

Initial ESP32 compass angles still come from `ingest.node_angles`; `0` is the top/north side of the rendered tank, `90` is right/east, `180` is bottom/south, and `270` is left/west. Once saved from the dashboard, `config/tank_layout.json` becomes the editable source for tank shape, camera placement, host association, and internals.

For permanent boot startup, use the template in `systemd/sync-tank-ingest.service`.

## Configure

Edit `config/sync_tank.yaml`.

For per-Pi identity, edit `config/tank_identity.yaml` first.

- `arm.servos`: GPIO pins, angle limits, neutral angle, pulse width.
- `cameras.esp32.discovery_ranges`: LAN ranges to scan for floater cameras.
- `hub`: main hub computer URL, API key, and forwarding mode.

Hub/VLM processing is intentionally not run on the Pi. The Pi sends frames to:

```text
POST {hub.base_url}/api/sync-tank/frame
```

and stream registrations to:

```text
POST {hub.base_url}/api/sync-tank/stream
```

## Hardware Notes

The default arm backend is direct GPIO PWM through `gpiozero` and `lgpio`. If GPIO cannot initialize, the app falls back to a mock driver so the dashboard and API can still be tested safely.

USB cameras are detected from `/dev/video*`. If no USB cameras are attached, the dashboard stays usable and shows an empty camera state.

ESP32-S3 cameras are discovered by probing common HTTP camera endpoints such as `/stream`, `/capture`, `/jpg`, and `/video`.
